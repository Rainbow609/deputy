# Deputy 重构技术设计文档

## 概述

本文档详细说明基于 mihomo-config 架构完全重构 deputy 项目的技术设计方案。重构目标包括：TOML 配置格式、GitHub Releases 发布机制、完整节点验证系统和单一配置输出。

## 架构设计

### 整体架构

采用 mihomo-config 的成熟架构，包括传输层、验证层、渲染层和发布层四个核心组件：

```
┌─────────────────────────────────────────────────────────────┐
│                        配置输入层                            │
│                    nodes.toml 配置文件                       │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                        传输获取层                            │
│         fetch_transport.py (传输链路管理)                    │
│  CloudscraperTransport → RequestsTransport → CurlCffiTransport │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                        节点验证层                            │
│              TCP探测 → HTTP检查 → 延迟测试                   │
│                   (30并发，3秒超时)                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                        质量统计层                            │
│         存活率计算 → 区域分析 → 延迟统计 → 质量评分           │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                        模板渲染层                            │
│         config.template.yaml → _safe_format_map → config.yaml │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                        发布分发层                            │
│              GitHub Releases → 版本管理 → 发布说明            │
└─────────────────────────────────────────────────────────────┘
```

### 核心组件

#### 1. 传输获取层 (`fetch_transport.py`)

**组件职责**: 处理订阅源节点的获取，支持反爬虫机制和连接问题处理。

**核心类设计**:

- `TransportChain`: 传输链路协调器
  - 管理多个传输实例的轮询和 fallback
  - 实现重试逻辑和错误分类
  - 提供统一的获取接口

- `CloudscraperTransport`: Cloudflare challenge 处理
  - 专门处理 Cloudflare 保护页面
  - 自动识别和绕过 challenge
  - 作为首选传输方式

- `RequestsTransport`: 标准 HTTP 请求
  - 基于 requests 库的标准实现
  - 作为 Cloudscraper 的 fallback
  - 处理常规 HTTP 请求

- `CurlCffiTransport`: 浏览器指纹伪装
  - 模拟真实浏览器特征
  - 处理高级反爬虫机制
  - 作为最后的 fallback 选项

**设计要点**:
- 采用轮询策略而非顺序尝试，确保每个传输都有机会
- 自动检测和处理各类反爬虫挑战
- 详细的错误日志和重试机制

#### 2. 节点验证层

**组件职责**: 对获取的节点进行多层次验证，确保节点质量和可用性。

**验证流程**:

```
节点输入 → TCP连通性探测 → HTTP健康检查 → 延迟测试 → 质量评分 → 排序输出
   ↓           ↓                 ↓              ↓           ↓          ↓
 基础过滤    存活性验证        功能验证        性能评估    综合评价    优先级排序
```

**关键参数**:
- 并发数: 30
- 超时时间: 3 秒
- 重试次数: 0 (快速失败策略)
- 地址族: auto (自动选择 IPv4/IPv6)

**验证逻辑**:

1. **TCP 连通性探测**:
   - 检查 server:port 的 TCP 连接可达性
   - 支持多地址解析和重试
   - 记录连接时间和失败原因

2. **HTTP 健康检查**:
   - 通过 HTTP 请求验证节点功能完整性
   - 检查响应状态和内容
   - 验证代理转发功能

3. **延迟测试**:
   - 测量节点的往返延迟
   - 多次测量取平均值
   - 用于节点排序和质量评估

#### 3. 质量统计层

**组件职责**: 生成节点的质量统计数据和报告。

**统计指标**:

- **存活率**: 总体存活率和各订阅源存活率
- **区域分布**: 各地区节点数量和存活情况
- **延迟统计**: 平均延迟、延迟分布、极值节点
- **质量评分**: 基于验证结果的综合质量分数
- **变更追踪**: 新增节点、失效节点、性能变化

**输出格式**:
- 控制台输出: 人类可读的统计报告
- JSON 输出: 机器处理的结构化数据
- 发布说明: GitHub Releases 专用格式

#### 4. 模板渲染层

**组件职责**: 使用 mihomo-config 的模板系统生成最终配置文件。

**模板设计**:

直接使用 mihomo-config 的 `config.template.yaml`，包含以下占位符:

- `{LOCAL_PROXIES}`: 静态节点配置块
- `{SUB_PROXIES}`: 订阅节点配置块
- `{PROXIES}`: 所有节点配置块
- `{NODE_SELECT_LIST}`: 节点选择列表
- `{DIALER_LIST}`: 拨号节点列表

**渲染引擎**:

移植 mihomo-config 的 `_safe_format_map` 函数:

```python
_PLACEHOLDER_PATTERN = re.compile(r"\{([A-Z_][A-Z0-9_]*)\}")

def _safe_format_map(template_text, mapping):
    """只替换 {KEY}，KEY 必须在 mapping 中；其他 {X} 保留原样"""
    def repl(match):
        key = match.group(1)
        if key in mapping:
            return str(mapping[key])
        return match.group(0)  # 保留不匹配的占位符
    return _PLACEHOLDER_PATTERN.sub(repl, template_text)
```

**输出结果**:
- 单一配置文件: `config.yaml`
- 包含完整的节点配置、代理组、规则等
- 格式符合 Clash Meta 规范

#### 5. 发布分发层

**组件职责**: 将生成的配置文件发布到 GitHub Releases。

**版本管理**:

- 版本号格式: `vYYYY-MM-DD-HHMMSS` (UTC 时区)
- 版本唯一性: 基于精确时间戳确保不重复
- 发布条件: 仅在配置有实际变更时发布

**发布内容**:

- 主文件: `config.yaml`
- 发布说明: 包含节点统计、变更信息、质量指标
- 元数据: 版本信息、生成时间、处理统计

**失败处理**:

- 部分失败策略: 某些订阅源失败不影响整体发布
- 错误报告: 在发布说明中标注失败源和原因
- 回滚机制: 保留历史版本，支持快速回退

## GitHub Actions 工作流设计

### 触发条件

- **定时触发**: 每 3 小时 (`cron: '0 */3 * * *'`)
- **手动触发**: `workflow_dispatch`
- **变更触发**: 脚本或配置文件变更时自动触发

### 工作流程

1. **环境准备**: 设置 Python 环境，安装依赖
2. **配置加载**: 读取并解析 `nodes.toml`
3. **节点获取**: 并发获取订阅源节点
4. **节点验证**: TCP、HTTP、延迟多层次验证
5. **质量统计**: 生成统计报告和质量指标
6. **配置渲染**: 使用模板生成 `config.yaml`
7. **变更检测**: 比较新旧配置，判断是否需要发布
8. **发布创建**: 创建 GitHub Release，上传配置文件
9. **结果通知**: 输出处理结果和统计信息

### 权限需求

- `contents: write`: 用于创建 Releases 和上传文件
- `GITHUB_TOKEN`: 用于 GitHub API 调用

## 技术风险和缓解措施

### 风险 1: 大规模节点验证的性能影响

**风险描述**: 大量节点的并发验证可能导致性能问题或超时。

**缓解措施**:
- 严格的并发控制 (30 并发)
- 合理的超时设置 (3 秒)
- 失败快速终止机制
- 详细的性能监控

**监控指标**:
- 验证总执行时间
- 内存使用情况
- CPU 占用率
- 各阶段耗时分布

### 风险 2: 传输链路稳定性

**风险描述**: 第三方传输库可能存在兼容性或稳定性问题。

**缓解措施**:
- 三层 fallback 机制 (cloudscraper → requests → curl_cffi)
- 详细的错误分类和日志
- 传输成功率监控
- 自动重试和错误恢复

**监控指标**:
- 各传输方式的成功率
- Fallback 触发频率
- 常见错误类型分布

### 风险 3: GitHub Actions 执行时间限制

**风险描述**: 完整的验证和发布流程可能超出 GitHub Actions 的时间限制。

**缓解措施**:
- 优化验证逻辑，减少不必要的操作
- 设置合理的超时和分阶段处理
- 监控各阶段执行时间
- 必要时拆分工作流

**监控指标**:
- 工作流总执行时间
- 各关键阶段耗时
- 超时发生频率

## 实现注意事项

### 依赖管理

使用 `uv` 作为 Python 包管理工具，主要依赖包括：

```toml
[dependencies]
PyYAML = "*"
requests = "*"
cloudscraper = "*"
curl-cffi = "*"

[dev-dependencies]
pytest = "*"
pytest-cov = "*"
```

### 错误处理

- 所有网络操作都需要异常处理
- 提供清晰的错误消息和日志
- 实现适当的重试和 fallback 机制
- 确保部分失败不影响整体流程

### 日志记录

- 详细记录关键操作和决策点
- 区分不同日志级别 (DEBUG, INFO, WARNING, ERROR)
- 包含足够的上下文信息用于调试
- 考虑日志轮转和存储管理

### 性能优化

- 合理使用并发和异步处理
- 避免不必要的重复操作
- 实现适当的缓存机制
- 监控和优化关键路径性能

### 安全考虑

- 验证所有外部输入
- 避免硬编码敏感信息
- 使用安全的 HTTP 连接
- 定期更新依赖包
